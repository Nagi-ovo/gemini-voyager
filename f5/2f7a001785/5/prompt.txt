# ChangeLog Modal Feature Implementation Plan

## Context

Add a version-based changelog popup that shows update announcements when users upgrade to a new version. The popup displays markdown content, supports i18n, and only shows once per version.

## Architecture Decisions

### i18n Strategy for Changelog Content
To minimize maintenance burden (user explicitly asked to avoid too many md files), we use a **single markdown file per version with i18n section headers**. Each markdown file contains all languages in one file using `<!-- lang:xx -->` markers. A parser extracts the section matching the user's language (fallback to `en`).

Example: `src/pages/content/changelog/notes/1.2.8.md`:
```md
<!-- lang:en -->
### What's New
- Feature A
- Bug fix B

<!-- lang:zh -->
### 更新内容
- 功能 A
- 修复 B
```

This way: **1 version = 1 file**, easy to maintain.

### Version Dismissed Storage
Use `chrome.storage.local` with key `gvChangelogDismissedVersion` storing the version string the user last dismissed. On each load, compare current extension version vs dismissed version — show modal only if they differ.

## Files to Create

1. **`src/pages/content/changelog/index.ts`** — Main module (init, version check, DOM injection, cleanup)
2. **`src/pages/content/changelog/notes/1.2.8.md`** — First changelog (template/placeholder for next release)

## Files to Modify

## Implementation Details

### 1. `src/pages/content/changelog/index.ts`

```typescript
export function startChangelog(): () => void
```

**Logic flow:**
1. Read `gvChangelogDismissedVersion` from `chrome.storage.local`
2. Compare with `EXTENSION_VERSION` from `src/core/utils/version.ts`
3. If same → return (already dismissed for this version)
4. Dynamically import the matching markdown file: `./notes/${EXTENSION_VERSION}.md`
   - If file doesn't exist (import fails) → return silently (no changelog for this version)
5. Parse the markdown to extract the section for the current language (using `cachedLanguage` or `getCurrentLanguage()`)
6. Render markdown to HTML using `marked` (already a dependency)
7. Sanitize with `DOMPurify` (already a dependency)
8. Inject overlay + dialog into DOM
9. On close button click → save `EXTENSION_VERSION` to `gvChangelogDismissedVersion` and remove DOM elements
10. Return cleanup function

**Markdown parsing helper:**
```typescript
function extractLocalizedContent(raw: string, lang: AppLanguage): string
```
Splits by `<!-- lang:xx -->` markers, returns matching section or falls back to `en`.

**Dialog footer:** includes i18n link to docs:
```
https://voyager.nagi.fun/{locale}/guide/getting-started
```
Where `{locale}` is the current language (`zh` is root locale → no prefix, others use `/{locale}/` prefix).

- Dark mode support via `html.dark`, `[data-theme='dark']`, `[data-color-scheme='dark']`

### 6. Markdown Note File Convention

Path: `src/pages/content/changelog/notes/{version}.md`

**Meta block** at the top of each file defines shared assets (images/GIFs) that are language-independent. Use a YAML-like front matter:

```md
---
images:
  hero: ./assets/1.2.8-hero.gif
  feature1: ./assets/1.2.8-feature1.png
---

<!-- lang:en -->
### What's New in v1.2.8

![hero](hero)

- Feature description

![feature1](feature1)

- Bug fix description

<!-- lang:zh -->
### v1.2.8 更新内容

![hero](hero)

- 功能描述

![feature1](feature1)

- 修复描述

<!-- lang:zh_TW -->
...
<!-- lang:ja -->
...
(repeat for all 10 languages)
```

**Image resolution:** During parsing, `![alt](key)` placeholders are resolved against the `images` map from meta. The actual image files are stored in `src/pages/content/changelog/notes/assets/` and imported via Vite's `import.meta.glob('./notes/assets/*', { eager: true, import: 'default' })` to get their resolved URLs at build time.

Asset directory: `src/pages/content/changelog/notes/assets/` — stores PNG/GIF files shared across all languages.

Vite raw import for markdown: `import.meta.glob('./notes/*.md', { query: '?raw', import: 'default', eager: false })` to dynamically load by version.
